    • 1.7.14
      April 2026
      • Multi-select bulk actions on every library — every row in the file list grew a checkbox for users with edit authority, plus a "Select all" checkbox and live count above the list. Per-row checkboxes only render where User.can_bulk_edit_categories(reading) returns True (which mirrors can_delete_reading), so an Editor in a library full of admin-uploaded files literally can't multi-select rows that aren't theirs. The bar's "Select all" only toggles authorized rows; the indeterminate state engages when the selection is partial.
      • Bulk edit categories — new "Edit categories" action opens a modal with three modes: Add (keep existing tags + add the picked ones), Remove (drop the picked ones from each row), and Replace (replace existing tags entirely with the picked ones). Posts to /libraries/<lid>/readings/bulk-categories. Per-row authorization re-fires server-side, so a tampered POST with admin-uploaded ids from an Editor session is silently filtered. Replace-with-empty is blocked outright on Intergroup libraries with the categories-required toggle on, so a single click can't strip the upload-time invariant.
      • Bulk delete — new "Delete" action (red button) wipes the selected readings via /libraries/<lid>/readings/bulk-delete. JS confirms with the count ("Delete N files? This can't be undone."), then submits a synthetic POST form with the CSRF token and one hidden input per id. Server enforces can_delete_reading per row and reports both deleted + skipped counts so editors see when authorization scoped their action down. Stored files + thumbnails are cleaned up via _delete_upload before each row is removed.
      • New can_bulk_edit_categories permission gate — per-reading authorization for the multi-select bulk-edit surface. Aliased to can_delete_reading: if you can destroy a row, you can mass-tag it. This intentionally pulls frontend_editors out of the bulk surface too — they can edit individual readings but per the role-permission spec they have no library-file deletion authority, so a single "destructive enough" gate covers both bulk operations.
      • Categories now work on every library — the Intergroup-only gate on the categories editor and the upload-modal picker was lifted. Admins manage categories on any library via the same fieldset on the library-edit modal; uploads in any library that has categories defined render the picker. The Require at least one category per upload toggle still only enforces on Intergroup libraries (matching the existing Intergroup filter UI's invariant); on regular libraries the picker is always optional.
      • Bulk action buttons appear only when something is checked — the "Edit categories" and "Delete" buttons hide entirely while the selection set is empty, instead of rendering as greyed-out disabled controls. Implemented as a CSS rule that sets display: none on disabled bulk-action buttons; the existing JS keeps managing the disabled state.
      • Select-all checkbox aligns with the per-row column — the bulk-action bar's left padding was reduced to 6px so its "Select all" checkbox sits at the same x-coordinate as every per-row checkbox below it (matching the row's li 2px + label 4px offset). Right padding (14px) preserved so the action buttons keep their breathing room from the bar edge.
      • _library_browse_url helper — internal redirect helper that resolves the canonical detail URL for a library (slug-based for Intergroup rows, id-based for everything else). Used by the new bulk-action endpoints so flash + redirect lands at the human-readable URL on Intergroup libraries instead of triggering an extra 301 hop through the id route.
