    • 1.7.3
      April 2026
      • Root URL behaves sensibly when the public frontend is off — visiting the bare domain (e.g. https://ts.dccma.com) used to render a 404 for signed-in non-editor users when the Web Frontend module was disabled, and bounced them to the login screen they'd already passed through when only public visibility was off. Both surfaces now redirect any signed-in visitor to the /tspro/ dashboard instead. Anonymous visitors still get the login screen; admins / frontend editors still see the preview render when the module is on but visibility is off.
    • 1.7.2
      April 2026
      • Zoom Tech images render again — Zoom Tech rich-content blocks (and any other content) saved with image src values like /uploads/<stored> went 404 after the 1.6.0 release moved the admin under /tspro: the legacy URLs resolved at the root, where there was no route. Added a back-compat alias on the public blueprint at /uploads/<path:stored> that hands off to the same send_from_directory(UPLOAD_FOLDER, stored) call as /tspro/uploads/<stored>, with the same @login_required guard so anonymous visitors can't probe upload filenames. No data migration needed — old saved content just works.
      • Apple touch icons — bundled apple-touch-icon_tspro.png and apple-touch-icon_dccma.png under app/static/img/ and wired them into the right templates so adding the site to an iOS / iPadOS home screen picks the correct icon: the tspro icon shows on every /tspro admin page, the login screen, and the setup wizard; the dccma icon shows on every public web frontend page (homepage, meeting detail, event detail, etc.). Both at the standard 180×180 size.
    • 1.7.1
      April 2026
      • Reusable detail-page Templates — new Web Frontend → Templates admin section (under Structure) is the home for all reusable content-type templates. Ships four meeting variants — Classic, Card Stack, Magazine, Minimal — and four event variants — Classic, Poster, Timeline, Minimal. Picker uses CSS-drawn thumbnails per template and previews the active selection inline. Saves into two new SiteSetting columns (frontend_meeting_template, frontend_event_template) and applies to every meeting / event detail page on the public site. Conceptually distinct from layouts: layouts control the block sequence of one specific page; templates here control how every page of a content type is rendered.
      • Per-template appearance overrides — every template card has a Customize panel that overrides the page's design tokens for just that template. Background color (toggle + picker), heading + body font (vendored Inter / Fraunces or any uploaded custom font), heading + body size in rem with 0.1 rem increments and a "Use template default" toggle per slider. Stored as JSON in SiteSetting.frontend_template_settings_json, keyed by content type + template key, emitted as inline CSS variables on the section so empty values fall through to the global Design page.
      • Mobile-aware fluid sizing — when a heading/body size override is active it's emitted as clamp(min, calc(offset + slope×vw), max) that interpolates linearly between 320 px (mobile floor) and 1200 px (the admin's desktop target). A 4 rem desktop heading shrinks to ~2.1 rem on a 375 px phone, with an absolute floor (1.25 rem heading / 1 rem body) so text never collapses. Floor is auto-clamped to never exceed the override value, so small overrides emit a constant rem instead of a degenerate clamp.
      • Buttons + links wired to design tokens — every CTA in the new templates renders through the global .fe-btn / .fe-btn-primary / .fe-btn-ghost system, so primary/secondary background + text + radius + decoration come from Design → Buttons. Surfaces, text, borders, brand accents resolve through --fe-color-* design tokens with raw vars as fallback. Inline non-button links use --fe-color-link + --fe-link-decoration.
      • Meeting logo on the public site — every meeting template now renders meeting.logo_filename at 180 px width via a new public route GET /pub/meeting-logo/<id>. Classic places it in a flex header column to the left of the type pill / heading / description; the column collapses entirely when no logo is uploaded, and the header content flushes flush-left with no reserved gutter. The backend meeting detail page bumps its own logo display to 100 px (admin meetings list keeps the existing 48 px inline thumbnail).
      • Public alert message — new Meeting.public_alert_message column shown on every meeting template in the same styled banner that used to render alert_message. The legacy alert_message is now labeled Backend Alert Message in the meeting edit modal and only appears on the admin meeting detail page; the new Public Alert Message field has its own row with help text explaining where it shows.
      • Per-file public visibility — files, external links, scripts, videos, images, and readings can be individually opted-in to the public meeting page. Inline Public toggle on every file row in the meeting edit modal flips MeetingFile.public_visible via a JSON endpoint without closing the modal; rolls back the visual state on request failure. Mirrored toggle inside each file's inline edit form. Each library row has a dashed-border sub-panel listing every reading with a "Show on public meeting page" checkbox — independent of the existing All / Granular backend mode.
      • Files & Readings block on every template — new public block emitted only when something is marked public, placed after Zoom/Location: in the Classic grid as the last card, in Card Stack between actions and prose, in Magazine's sticky sidebar, and after Minimal's info list. Files are grouped by category (Documents, Scripts, Readings, Links, Videos, Images); public readings live in their own group.
      • Pretty resource URLs — public files and readings now resolve at /meeting/<meeting-slug>/<filename>.ext (e.g. /meeting/daily-zoom-round-up/opening-statement.pdf). Lookup is by id-equivalent slug match against the meeting's public files first, then meeting.public_readings. url_slug derives from the original filename when present (preserving the extension), falling back to the slugified title otherwise.
      • External links go straight to their destination — anchors for external_links and videos categories (and URL-only readings) use the raw f.url as the href instead of routing through the meeting-resource handler, so the browser's status bar shows the real destination on hover and right-click → Copy Link copies the actual URL. Uploaded files still go through the slug route so we can serve the stored upload with its original download name.
      • All resource links open in a new tab — every link in the public Files & Readings block now opens with target="_blank" rel="noopener" and shows the ↗ indicator, regardless of whether it's an external URL, a stored upload, or rendered body content.

    • 1.6.2
      April 2026
      • Update-available banner now notices same-version redeploys — the client-side poller used to compare only __version__ strings, so pushing a new image under the same version went unnoticed. app/version.py now also computes a __build_id__ at import time: a truncated blake2b hash of every .py, .html, .css, and .js file under app/. Plain container restarts with unchanged code keep the same hash and don't flap the banner.
