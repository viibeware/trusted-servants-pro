    • 1.9.1
      May 2026
      • Frontend export/import now produces a verbatim 1-to-1 copy (bundle v3 → v4) — two gaps in the frontend bundle were silently dropping per-page customisation on every restore. (1) Per-page spacing columns (pad_top, pad_bottom, pad_x, section_gap, block_margin_y, defaults 80/96/16/32/12) were previously not exported and not imported — every restored page reverted to the model's defaults. Both sides now carry them; the import side falls back to the same defaults via a local _opt_int helper when a v3 bundle predates the change, so older bundles continue to import without surprises. (2) Homepage designation — SiteSetting.homepage_page_id is a Page FK, and page IDs aren't portable across installs. Export now resolves the FK → page slug as settings.homepage_page_slug in the payload; import side runs after pages are restored, looks up the page by slug, and writes the new id back to the column. When the slug is missing (v3 bundle) or doesn't match any imported page, the destination keeps whatever its own auto-seed wrote — / stays 200 either way. Manifest scope note rewritten to call out the v4 additions and the v3 backwards-compat path; format version bumped in both manifest.json and the inner frontend.json payload's format_version constant. No code changes needed for the rest of the v1.9.0 surface area that lives inside blocks_json — Container per-side borders, hover border-width, mobile direction/padding, height/min-height, dark-mode colours; Hero dark-mode gradient + sub colour; Features / FAQ heading + subheading + items list; Meetings / Events filter + display toggles. These all ride along byte-for-byte. The whole-site bundle (verbatim SQLite copy) likewise needs no changes — homepage_page_id rides along with the raw .db file.
    • 1.9.0
      May 2026
      • Homepage is now a Page (legacy homepage admin + public renderer retired) — the public / root is driven by whichever Page row the admin designates as the homepage. The existing inputs were relabelled (light) so the pairs read clearly. (1) @media (max-width: 640px) .fe-pp { padding: 56px 0 72px } discarded the inline --fe-pp-pad-top / --fe-pp-pad-bottom variables — an admin who set pad_top: 0 from the page-settings panel still saw 56px on phones. (2) @media (max-width: 640px) .fe-pp-section { margin-top: 24px } pinned every section after the first to 24px regardless of the admin's section_gap. Removed — desktop's var(--fe-pp-section-gap, 32px) carries through, so section_gap: 0 produces zero margin on mobile too.

Per-page edit modals are included AFTER the icon picker in frontend_page_edit.html, so the picker was getting layered behind.
